## Context

当前项目的核心玩法与经济系统已经完整，主界面包含 2D/3D 两套展示，其中 3D 场景是主要游玩入口。现有画面具备基础网格、蛇体和食物模型，但关键状态变化缺乏视觉反馈，导致玩家在高速移动时不易感知事件（进食、护盾触发、危险状态）。

约束：
- 不改变游戏规则、碰撞逻辑和商店数值。
- 不新增外部渲染库，沿用 React + @react-three/fiber + three。
- 在桌面和移动端均可保持可玩帧率。

## Goals / Non-Goals

**Goals:**
- 增强 3D 场景层次感（背景、地面、边缘细节），让画面不再“平”。
- 为蛇、食物与关键事件提供可观察的动态反馈。
- 在 HUD 提供即时信息提示（最近事件与状态高亮），减少只靠文本数字理解状态的负担。

**Non-Goals:**
- 不重构核心状态机与经济系统。
- 不引入复杂后处理（post-processing）或重型粒子系统。
- 不调整地图大小、移动规则、碰撞判定与道具价格。

## Decisions

1. 使用轻量级程序化细节而非新依赖
- 选择：使用 R3F 基础几何体、材质和简单动画（`useFrame`）实现粒子和脉冲反馈。
- 原因：实现成本低、可控、无需引入额外依赖和构建风险。
- 备选：引入后处理/特效库。未采用，因为超出需求且增加性能与维护负担。

2. 事件反馈采用短时状态驱动
- 选择：为“进食”“护盾触发”维护短时视觉状态（例如 300~500ms），驱动 UI 徽标与场景波纹。
- 原因：避免复杂时间线，同时可复用 React 状态管理。
- 备选：全局动画时间轴。未采用，因为与现有代码风格不一致且过重。

3. 统一在现有 `App.jsx` 扩展而不拆分大型架构
- 选择：在保持现有组件结构下新增小型视觉组件（如 `BoardAura`、`StarField`、`EventBadge`）。
- 原因：本次改动聚焦表现层，局部扩展能降低回归风险。
- 备选：重构为多文件场景系统。未采用，因为超出 issue 范围。

## Risks / Trade-offs

- [Risk] 动画细节过多可能影响低端设备帧率 → Mitigation: 限制粒子数量、降低几何复杂度、避免每帧创建对象。
- [Risk] 新增视觉状态与游戏状态同步错误 → Mitigation: 将事件状态集中到单一来源并用短时定时器自动回收。
- [Risk] HUD 信息过密降低可读性 → Mitigation: 仅保留高价值提示（最近事件、护盾状态），避免堆叠文本。
